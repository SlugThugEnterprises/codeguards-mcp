# CodeGuards v0.2 Architecture Spec

CodeGuards is an MCP-based architecture enforcement system that converts AI development intent into a persistent, machine-enforceable contract and continuously ensures generated code adheres to it.

This document defines the v0.2 mental model, file contracts, plugin standard, and runtime guarantees. The goal is to keep the project coherent as it evolves from useful guards into a contract-driven execution loop for AI software development.

---

## One-line definition

CodeGuards is an MCP-based system that converts AI coding intent into an enforceable architecture contract and continuously prevents code from drifting away from that contract.

---

## System model

```text
Intent
  what are we building?
        ↓
Architecture
  how is it structured?
        ↓
Plan
  how do we implement it?
        ↓
Code generation
        ↓
Enforcement
  do the changes still match the contract?
        ↓
Drift correction
  AI fixes issues and continues
        ↺
```

This loop is the product. Everything else is infrastructure.

---

## System layers

CodeGuards is split into four conceptual layers.

| Layer | Purpose | Primary artifact/tool |
|---|---|---|
| Intent Layer | What are we building? | `.codeguards/intent.json` |
| Architecture Layer | How is it structured? | `.planning/ARCHITECTURE.md` |
| Planning Layer | How do we implement it? | `.planning/PROJECT_PLAN.md` |
| Enforcement Layer | Does code still match intent? | `check_project`, `check_file` |

## 1. Intent Layer

### File

```text
.codeguards/intent.json
```

### Purpose

The intent layer captures decisions, not code.

Examples:

- architecture style: layered, hexagonal, modular, event-driven
- domain boundaries
- error-handling strategy
- logging and tracing strategy
- testing expectations
- dependency philosophy
- project maturity: prototype, internal tool, production system, library

### Rule

If intent is missing, full-project enforcement is not allowed.

This is the first philosophical boundary in the system: no architecture-less execution.

### Draft schema

```json
{
  "project_name": "string",
  "goal": "string",
  "architecture_style": "layered | hexagonal | modular | event-driven | other",
  "stage": "prototype | internal | production | library",
  "modules": [
    {
      "name": "string",
      "responsibility": "string"
    }
  ],
  "rules": {
    "error_handling": "string",
    "logging": "string",
    "testing": "string",
    "observability": "string"
  },
  "constraints": {
    "allow_todos": false,
    "allow_debug_prints": false,
    "max_complexity": "low | medium | high"
  }
}
```

## 2. Architecture Layer

### File

```text
.planning/ARCHITECTURE.md
```

### Format

Markdown with YAML frontmatter.

The markdown body is for humans. The YAML frontmatter is for CodeGuards.

### Purpose

The architecture layer compiles intent into an architecture contract:

- module boundaries
- allowed dependencies
- structural constraints
- responsibility definitions
- architectural decisions
- quality goals that affect structure

### Rule

`ARCHITECTURE.md` is the canonical architecture contract **and the scope gate**.

It is not passive documentation. It defines what must remain true unless the contract is deliberately updated. Its presence or absence determines which tests run — no directory-based scoping needed.

### Draft YAML shape

```yaml
---
project:
  name: example-api
  type: web_api
  stage: production

architecture_profile:
  name: layered
  rationale: Keep request handling, business workflows, and persistence separate.

modules:
  - name: api
    path: src/api/
    responsibility: HTTP routing and request/response translation only.
  - name: application
    path: src/application/
    responsibility: Business workflows and orchestration.
  - name: domain
    path: src/domain/
    responsibility: Domain model and rules.
  - name: infrastructure
    path: src/infrastructure/
    responsibility: Database, network, and external integration implementations.

allowed_dependencies:
  api:
    - application
  application:
    - domain
    - ports
  domain: []
  infrastructure:
    - domain
    - ports

constraints:
  no_direct_database_from_api:
    enabled: true
    decision: Request handlers must go through application services.
  domain_no_framework_imports:
    enabled: true
    decision: Domain code must stay portable and testable.
  structured_observability:
    enabled: true
    decision: Production workflows require traceable execution paths.

quality_goals:
  tests:
    unit_required: true
    integration_required_for:
      - persistence
      - external_api_clients
  docs:
    public_api_docs_required: true
---

# Architecture

Human-readable architecture explanation goes here.
```

## 4. Enforcement Layer

### Philosophy: Coach, not Cop

The enforcement layer is designed to **help the AI succeed**, not to police it. Every interaction should feel like a copilot offering support, not a referee calling penalties.

Key principles:
- **Positive reinforcement** — report what's going well, not just violations
- **Opt-in checkpoints** — the agent can skip with explanation, not hard blocks
- **Collaborative resolution** — violations offer paths forward, not just verdicts
- **Statistical sampling** — random checkpoints prevent gaming without feeling like gotchas
- **Agent-tunable** — the system adapts to the agent's patterns over time

### Tools

```text
check_project
check_file
```

### Behavior

The enforcement layer:

1. checks for `.planning/ARCHITECTURE.md`
2. if present: loads architecture, resolves applicable tests from `tests.yaml`
3. if absent: runs only baseline hygiene tests (credentials, unsafe patterns, no stubs)
4. fetches test modules from `~/.codeguard/tests/`
5. runs tests against the project
6. returns a **coaching report** with violations, positives, and suggestions

### Coaching Report Shape

```json
{
  "summary": {
    "files_checked": 12,
    "violations": 2,
    "positives": 5,
    "health_score": 85
  },
  "positives": [
    {
      "guard": "doc_comments",
      "message": "All 12 public functions have doc comments — great documentation discipline",
      "streak": 8
    },
    {
      "guard": "error_handling",
      "message": "Error types are consistent across all modules",
      "streak": 3
    }
  ],
  "violations": [
    {
      "guard": "test_entry_point_init",
      "severity": "warning",
      "file": "crates/adcore-api/src/main.rs",
      "line": 1,
      "message": "Entry point missing logging::init call",
      "suggestion": "Add logging::init(&config.log_level) before rt.block_on(), or explain why it's not needed here",
      "can_skip": true,
      "skip_requires_reason": true,
      "contract_path": ".planning/ARCHITECTURE.md",
      "contract_section": "constraints.structured_observability"
    }
  ],
  "suggestions": [
    "You've skipped review on 3 test files in a row. Want to lower the sampling rate for tests?",
    "Your error handling is consistent — consider extracting a shared error module"
  ]
}
```

### Enforcement Modes

The user (or agent) selects an enforcement mode during the architecture interview:

| Mode | Behavior | Best for |
|---|---|---|
| `fast` | Plan+Write only. No checkpoints. Suggestions only. | Prototypes, throwaway code |
| `balanced` | Plan+Write + random sampling (2-5 file interval). Opt-in checkpoints with skip-able reviews. | Most projects |
| `strict` | Plan+Write+Read+Fix on every file. Required checkpoints. | Security-sensitive, production |

The agent can change modes mid-project: "This is getting complex — want to switch to strict mode?"

### Rule

Enforcement scope is determined by the presence of an architecture contract AND the selected mode.

- **With `ARCHITECTURE.md` + `strict` mode**: full enforcement with required checkpoints
- **With `ARCHITECTURE.md` + `balanced` mode**: sampling-based checkpoints, skip-able with reason
- **With `ARCHITECTURE.md` + `fast` mode**: suggestions only, no blocks
- **Without `ARCHITECTURE.md`**: baseline hygiene only + prompt to declare architecture

This is the second philosophical boundary: the architecture document is both the contract and the scope gate. No directory-based scoping is needed.

### Test Library Architecture (v0.3)

Guard tests live **outside the project** in `~/.codeguard/tests/`. This is critical: when tests are managed by the MCP server rather than the project codebase, the AI cannot silently edit them to make failures go away.

But crucially: the agent can **propose new tests** and **challenge existing ones**. When a test fires a violation, the agent can say "this test doesn't apply here because..." and the system learns. Over time, the test library becomes a collaboration between human intent and AI expertise.

## Core guarantees

These are non-negotiable system properties.

### 1. Contract-first execution

No `intent.json` means no full-project enforcement.

### 2. Architecture is authoritative

Once declared, architecture is:

- loaded,
- cached where useful,
- enforced consistently,
- not silently re-inferred.

Architecture can change, but only through an explicit contract update flow.

### 3. Generic-first operation

The system must work with:

- zero plugins,
- zero AST support,
- zero language-specific knowledge.

Plugins improve precision. They are not required for initial value.

### 4. Drift detection is primary

The primary value is preventing structural decay over time.

Style, lint, and hygiene checks are supporting pressure, not the product identity.

### 5. No implicit global opinion

Every enforcement rule must either:

- derive from intent or architecture, or
- be explicitly declared as baseline or plugin behavior.

This prevents CodeGuards from collapsing into opinionated linter chaos.

## Existing-project flow

```text
Existing repository
        ↓
inspect current structure
        ↓
bootstrap observed architecture
        ↓
ask user what should be preserved, cleaned up, or changed
        ↓
write intent + architecture contract
        ↓
enforce future changes against intended architecture
```

Existing projects require a distinction between observed architecture and intended architecture.

| Concept | Meaning |
|---|---|
| Observed architecture | What the code currently does |
| Intended architecture | What future work must preserve or move toward |

CodeGuards should not preserve accidental messes by default.

## ArchitectureContractGuard

The central v0.2 implementation target is a core `ArchitectureContractGuard`.

It should read `.planning/ARCHITECTURE.md` and enforce contract-derived constraints through one coherent guard, not by generating many unrelated one-off rule implementations.

```text
.planning/ARCHITECTURE.md
        ↓
ArchitectureContractGuard
        ↓
contract-derived constraints
        ↓
violations linked to decisions
```

### Responsibilities

The guard should evaluate:

- layer dependency rules,
- module ownership,
- forbidden cross-module access,
- domain/infrastructure separation,
- architecture-profile constraints,
- project-specific quality goals that affect structure.

### Non-responsibilities

The guard should not become:

- a full linter,
- a formatter,
- a security scanner,
- a type checker,
- a replacement for Clippy, Ruff, tests, or CI.

## Plugin standard v0.2

Plugins are an extension model for language-specific depth. They are not the architecture contract mechanism.

### Goals

Plugins must be:

- simple,
- safe,
- predictable,
- composable,
- loosely coupled to core.

### Structure

Each plugin is a Python module:

```text
plugins/<name>.py
```

Each plugin exposes:

```python
def register(registry: GuardRegistry) -> None:
    ...
```

### Plugin metadata

Each plugin should register metadata:

```python
registry.register_plugin_metadata({
    "id": "rust_fast_checks",
    "name": "Rust Fast Checks",
    "version": "1.0.0",
    "language": "rust",
    "speed": "fast",
    "scope": ["file"],
    "requires_ast": False,
})
```

### Guard registration

```python
registry.register_guard(
    name="no_unwrap",
    languages=["rust"],
    scope="file",
    severity="critical",
    check_fn=check_unwrap,
)
```

### Extractors

Extractors are the critical abstraction.

```python
registry.register_extractor(
    capability="function_blocks",
    file_extensions={".rs"},
    extractor_fn=parse_rust_functions,
)
```

Meaning:

- core does not need to parse every language,
- plugins optionally provide structure knowledge,
- generic guards degrade gracefully without extractors.

### Guard execution context

A future SDK should pass each guard a context object shaped like:

```json
{
  "file_path": "string",
  "source": "string",
  "intent": {},
  "architecture": {},
  "config": {}
}
```

### Guard return shape

```json
{
  "guard": "no_unwrap",
  "severity": "critical",
  "file": "src/main.rs",
  "line": 42,
  "message": "Use of unwrap() detected.",
  "fix": "Use the ? operator or return a typed error.",
  "intent_link": "error_handling"
}
```

### Capabilities

Plugins should declare capabilities such as:

```yaml
capabilities:
  - file_check
  - project_check
  - fix_suggestions
  - ast_required
```

### Safety rules

Plugins must not:

- access files outside the project root,
- mutate files directly,
- execute network calls unless explicitly allowed by a future policy,
- bypass the unified output schema.

### Performance model

| Scope | Trigger |
|---|---|
| file | per file change or per file scan |
| project | per `check_project` |
| structural | cached within a run |
| AST plugins | optional and lazy |

## Product definition

### What CodeGuards is

A lifecycle enforcement system for AI-generated software architecture.

A contract compiler plus enforcement runtime for AI-generated software architecture.

### What CodeGuards is not

- not a linter,
- not a formatter,
- not a static analyzer,
- not a security scanner,
- not a test runner,
- not a replacement for the existing toolchain.

### What CodeGuards does uniquely

- captures architecture before code exists,
- enforces architecture during code generation,
- corrects drift after violations appear,
- keeps AI aligned across long sessions,
- reduces friction with Clippy, Ruff, tests, CI, reviewers, and humans.
